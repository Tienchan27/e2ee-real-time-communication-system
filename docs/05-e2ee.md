# 05 - Kiến Trúc E2EE

## Mục tiêu

Chốt mô hình mã hóa đầu cuối ở mức thực tế cho phạm vi đồ án: đủ đúng về nguyên tắc bảo mật và đủ khả thi để triển khai.

## Ngoài phạm vi

- Full Signal Double Ratchet.
- Multi-device key sync.
- Advanced deniability guarantees.

## Mô hình bảo mật

- Plaintext chỉ tồn tại trên bộ nhớ client.
- Server chỉ lưu và relay ciphertext envelope.
- Metadata trao đổi khóa có thể nhìn thấy ở server.

## Thành phần mật mã

- Key agreement: `P-256` (ECDH; Web Crypto API browser-native — `crypto.subtle.generateKey / deriveBits / importKey`).
- Symmetric encryption: `AES-256-GCM`.
- KDF: `HKDF-SHA256`.
- Randomness: browser `crypto.getRandomValues`.

## Định dạng envelope

```txt
messageId: string(uuid-v7) [required]
conversationId: string(uuid-v7) [required]
senderDeviceId: string(uuid-v7) [required]
algorithm: string [required, aes-256-gcm]
keyVersion: number [required]
nonce: string(base64) [required, 96-bit]
ciphertext: string(base64) [required]
aad: object [optional, wire metadata; persisted in DB; not always used as GCM additionalData]
```

Trường `aad` trên envelope được persist và relay qua `chat:message` / GET messages. Với G-lite, `aad` mang `e2eeSetup: g-lite-v1` và ephemeral public key của người gửi — không bind vào AES-GCM AAD khi thiết lập khóa lần đầu.

Quy tắc bắt buộc:
- `nonce` không được tái sử dụng trong cùng `senderDeviceId + conversationId + keyVersion`.
- Nếu phát hiện nonce trùng, message phải bị từ chối và ghi nhận sự kiện bảo mật.

## Luồng trao đổi khóa

**Socket fallback** tại `frontend/src/crypto/socketKeyExchange.ts` (được gọi từ `ChatContext` khi G-lite prekey không khả dụng).

1. A tạo cặp khóa ephemeral P-256 (`crypto.subtle.generateKey`).
2. A export public key dạng SPKI (base64), gửi `key:exchange:init` kèm public key và `sessionProposalId`.
3. B nhận, tạo cặp khóa ephemeral riêng, dẫn xuất shared key, lưu vào `cryptoManager.setConversationKey(conversationId, 1, key)`, gửi `key:exchange:response` kèm public key của B.
4. A nhận response, dẫn xuất shared key từ private key A + public key B, lưu conversation key.
5. HKDF-SHA256: salt = `TextEncoder().encode(conversationId)`, info = `"e2ee-chat-v1"`, keyLen = 256-bit AES-GCM key.
6. Hai bên đặt `keyVersion=1`. Timeout initiation: 8 giây.

## G-lite v1 (first message khi peer offline)

**Đã triển khai** tại `frontend/src/crypto/deviceKey.ts`, `frontend/src/crypto/conversationKeys.ts`, `frontend/src/crypto/cryptoSession.ts`, API `PUT /devices/me/ecdh-public-key` và `GET /users/:userId/ecdh-public-key`.

### Lưu trữ khóa theo userId

- Device private key: `e2ee:device:ecdh:private:{userId}` (migrate một lần từ key global cũ `e2ee:device:ecdh:private`).
- Conversation key: `e2ee:key:{userId}:{conversationId}:{keyVersion}`.
- `setActiveCryptoUserId(userId)` được gọi khi login/logout/restore — cho phép test A/B luân phiên trên **cùng browser profile** mà không ghi đè khóa nhau.
- Khuyến nghị QA: dùng **2 browser profile** riêng khi test song song; scoped keys vẫn bắt buộc khi luân phiên login.

### Checklist prekey trước khi test G-lite

1. User B login → DevTools Network: `PUT .../ecdh-public-key` trả **200** (không 401).
2. B logout (hoặc đóng tab) → A gửi tin đầu.
3. B login lại cùng profile → mở chat → decrypt OK.

Luồng gửi tin đầu (A → B, B có thể offline):

1. Sau login, mỗi device upload ECDH public key (SPKI base64) lên server.
2. A gọi `ensureKeyForSend(conversationId, peerUserId)`:
   - Nếu đã có conversation key trong `localStorage` → dùng luôn.
   - Nếu chưa: `GET /users/:peerUserId/ecdh-public-key` → 404 → fallback socket key exchange (`PEER_PREKEY_MISSING`).
   - Nếu có prekey: A tạo ephemeral ECDH pair, `deriveSharedKey(A_ephemeral_private, B_static_public, conversationId)`, lưu AES key.
3. A mã hóa tin nhắn **không** bind setup metadata vào GCM `additionalData`.
4. Tin đầu tiên kèm `envelope.aad`:
   ```json
   { "e2eeSetup": "g-lite-v1", "senderEphemeralPublicKey": "<base64>", "senderDeviceId": "<uuid>" }
   ```
5. Realtime persist `aad` trong DB và relay qua `chat:message`.

Luồng nhận (B online hoặc sau khi login):

1. `loadMessages` gọi `ensureKeyFromGliteHistory` **trước** `joinConversation` và trước khi dùng khóa từ socket exchange — tránh ghi đè `K_glite` bằng `K_socket`.
2. Nếu lịch sử có G-lite nhưng derive thất bại: **không** fallback `loadConversationKey` (socket key cũ); xóa stale key và đánh dấu conv G-lite.
3. Từ `chat:message` hoặc `GET /messages`, nếu `aad.e2eeSetup === "g-lite-v1"`:
   - Import `senderEphemeralPublicKey`, `deriveSharedKey(B_static_private, A_ephemeral_public, conversationId)`.
   - Lưu conversation key (`keyVersion=1`) vào `localStorage` (scoped theo `userId`).
4. Nếu giải mã thất bại: xóa khóa hiện tại, derive lại từ `aad` (force), thử giải mã một lần nữa.
5. Khi conv đã có G-lite setup, **không** chạy socket key exchange (`shouldAllowSocketKeyExchange`).

Sender (tin đầu):

- Trước khi gửi lần đầu (chưa có tin `sent` từ mình): xóa conversation key cũ để tránh encrypt bằng key socket stale.
- Sau timeout / peer không có prekey: bubble `failed` + message rõ (không queue im lặng).

### Ma trận nghiệm thu G-lite

| # | Case | Kỳ vọng |
|---|------|---------|
| 1 | B login → logout → A gửi (B offline) → B login cùng profile | B decrypt OK |
| 2 | A/B cùng browser, luân phiên login (2 userId) | Case 1 vẫn OK |
| 3 | B chưa login bao giờ → A gửi | A bubble `failed` + message rõ |
| 4 | B online Home, chưa mở chat, no prekey, A gửi | Socket exchange OK hoặc failed rõ |
| 5 | Reload trang B sau decrypt OK | Tin vẫn đọc được |
| 6 | G-lite + reload regression | Không socket overwrite |
| 7 | Token gần hết hạn → reload B | Prekey upload + GET messages không 401 |

Giới hạn G-lite v1:

- Một active prekey mới nhất mỗi user (multi-device chưa hỗ trợ chọn đúng device).
- Forward secrecy yếu hơn pure ephemeral socket exchange (device static key tồn tại lâu).
- Đổi thiết bị / xóa site data / incognito → device private key mới ≠ prekey lúc A gửi → tin cũ không decrypt được; tin **mới** sau khi B online và upload prekey mới vẫn work (G-lite hoặc socket).
- `envelope.aad` là wire metadata; không dùng làm AES-GCM AAD khi thiết lập khóa lần đầu.

Khôi phục thủ công (chỉ khi device private còn khớp prekey lúc gửi):

1. Xóa `localStorage` key `e2ee:key:{userId}:{conversationId}:1` (hoặc legacy `e2ee:key:{conversationId}:1`).
2. Reload trang và mở lại cuộc trò chuyện — client derive lại từ `aad` trong lịch sử tin.
3. Nếu vẫn fail: prekey mismatch thật — nhờ người gửi gửi tin mới.

## Chính sách xoay khóa (bản tối thiểu — đã triển khai FE)

**Đã triển khai** tại `frontend/src/crypto/manager.ts`, `frontend/src/crypto/conversationKeys.ts`, `frontend/src/context/ChatContext.tsx`. Backend relay `key:rotate` + tie-break tại `realtime-service/src/socket/key.ts`.

### Trigger

Sau mỗi tin gửi thành công, initiator kiểm tra (không dùng timer nền):

- `N = 50` tin kể từ lần rotate gần nhất, hoặc
- `T = 30 phút` kể từ lần rotate gần nhất.

### Cơ chế ratchet đối xứng

`K_(n+1) = HKDF-SHA256(ikm = raw(K_n), salt = rotationSalt, info = "e2ee-rotate-v{n+1}")`.

Không cần handshake ECDH 2 chiều — không đụng luồng G-lite / socket key exchange ban đầu.

### Nguồn sự thật: `envelope.aad`

Tin **đầu tiên** sau rotate mang:

```json
{
  "e2eeRotate": "v1",
  "rotationSalt": "<base64 32 bytes>",
  "fromKeyVersion": 1,
  "toKeyVersion": 2
}
```

Peer offline mở lại lịch sử vẫn derive được key mới từ `aad` (giống pattern G-lite). `aad` là wire metadata, không bind vào AES-GCM `additionalData`.

### Socket `key:rotate` (peer online)

Initiator gửi `key:rotate` với `senderEphemeralPublicKey` = **cùng `rotationSalt`** như trong `aad` → peer online pre-derive trước khi nhận tin. Server tie-break khi rotate đồng thời (xem `03-events.md`).

### Lưu trữ & hiệu năng

- Mỗi `keyVersion` persist scoped theo `userId` (`e2ee:key:{userId}:{convId}:{version}`).
- Giữ tối đa **2 version** gần nhất trong memory + localStorage (prune khi rotate).
- `activeKeyVersion` sync sau `loadMessages` (max version trong lịch sử).

### Chưa làm (ngoài phạm vi bản tối thiểu)

- `key:rekey_required` recovery flow
- Rotate thủ công từ UI
- Grace timer 2 phút (chỉ prune theo version count)

## Khôi phục rekey

Điều kiện kích hoạt:
- Người nhận nhận được message có `keyVersion` không hợp lệ.

Các bước khôi phục:
1. Recipient emits `key:rekey_required`.
2. Sender pauses new sends for that conversation.
3. Both run fresh exchange.
4. Retry failed message with same `messageId` and new `requestId`.

## Xử lý lỗi

- Nếu giải mã thất bại:
  - hiển thị lỗi an toàn, không lộ nội dung.
  - chỉ gửi telemetry với metadata đã làm sạch.
- Nếu trao đổi khóa timeout:
  - retry tối đa 3 lần, sau đó đánh dấu conversation chưa sẵn sàng mã hóa.

## Giả định bảo mật

- Client không bị chiếm quyền bởi malware.
- TLS được cấu hình đúng.
- Flow OTP và đăng nhập không bị bypass.
- Browser crypto API được coi là primitive đáng tin cậy của nền tảng.

## Giới hạn hiện tại

- Chưa có mức post-compromise security tương đương ratchet protocol.
- Chưa có đồng bộ đa thiết bị đúng nghĩa.
- Bảo mật metadata còn giới hạn (đồ thị liên hệ, thời gian).

## Hướng nâng cấp

- Bổ sung per-message ratchet.
- Bổ sung device identity key và signed prekey.
- Bổ sung key transparency log.

## Trách nhiệm

- System Owner: chốt chính sách mật mã và tiêu chí nghiệm thu.
- Phụ trách FE: triển khai crypto phía client và lưu trữ khóa cục bộ.
- Phụ trách Realtime: đảm bảo vận chuyển sự kiện trao đổi khóa ổn định.
