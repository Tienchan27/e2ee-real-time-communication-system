# 05 - Kiến Trúc E2EE

## Mục tiêu

Chốt mô hình mã hóa đầu cuối ở mức thực tế cho phạm vi đồ án: đủ đúng về nguyên tắc bảo mật và đủ khả thi để triển khai.

## Ngoài phạm vi

- Full Signal Double Ratchet.
- Multi-device key sync.
- Advanced deniability guarantees.

## Mô hình bảo mật

- Plaintext chỉ tồn tại trên bộ nhớ client.
- Server chỉ lưu và relay ciphertext envelope.
- Metadata trao đổi khóa có thể nhìn thấy ở server.

## Thành phần mật mã

- Key agreement: `X25519` preferred (`P-256` fallback).
- Symmetric encryption: `AES-256-GCM`.
- KDF: `HKDF-SHA256`.
- Randomness: browser `crypto.getRandomValues`.

## Định dạng envelope

```txt
messageId: string(uuid) [required]
conversationId: string(uuid) [required]
senderDeviceId: string(uuid) [required]
algorithm: string [required, aes-256-gcm]
keyVersion: number [required]
nonce: string(base64) [required, 96-bit]
ciphertext: string(base64) [required]
aad: object [optional, includes conversationId and timestamp]
```

## Luồng trao đổi khóa

1. A generates ephemeral key pair.
2. A emits `key:exchange:init` with public key.
3. B generates ephemeral key pair and emits `key:exchange:response`.
4. Both derive shared secret via ECDH.
5. Both derive session key using HKDF with same salt/context.
6. Both set `keyVersion=1`.

## Chính sách xoay khóa (Hybrid)

Xoay khóa khi đạt một trong hai điều kiện:

- Số lượng tin nhắn đạt ngưỡng: `N=50`.
- Tuổi khóa đạt ngưỡng: `T=30 phút`.

Luồng xoay khóa:

1. Triggering client emits `key:rotate` with new ephemeral pubkey.
2. Peer derives new key and updates local state.
3. Both set `keyVersion = previous + 1`.
4. Keep previous key in short grace window for in-flight decrypt (max 2 minutes).

## Khôi phục rekey

Điều kiện kích hoạt:
- Người nhận nhận được message có `keyVersion` không hợp lệ.

Các bước khôi phục:
1. Recipient emits `key:rekey_required`.
2. Sender pauses new sends for that conversation.
3. Both run fresh exchange.
4. Retry failed message with same `messageId` and new `requestId`.

## Xử lý lỗi

- Nếu giải mã thất bại:
  - hiển thị lỗi an toàn, không lộ nội dung.
  - chỉ gửi telemetry với metadata đã làm sạch.
- Nếu trao đổi khóa timeout:
  - retry tối đa 3 lần, sau đó đánh dấu conversation chưa sẵn sàng mã hóa.

## Giả định bảo mật

- Clients are not malware-compromised.
- TLS is correctly configured.
- OTP and credential flow not bypassed.
- Browser crypto APIs are trusted platform primitives.

## Giới hạn hiện tại

- No post-compromise security equivalent to ratchet protocols.
- No true multi-device consistency.
- Metadata privacy is limited (contact graph, timing).

## Hướng nâng cấp

- Add per-message ratchet.
- Add device identity keys and signed prekeys.
- Add key transparency log.

## Trách nhiệm

- System Owner: chốt chính sách mật mã và tiêu chí nghiệm thu.
- Phụ trách FE: triển khai crypto phía client và lưu trữ khóa cục bộ.
- Phụ trách Realtime: đảm bảo vận chuyển sự kiện trao đổi khóa ổn định.
